Automation Tool

GitHub Actions

GitHub Actionsは、GitHub上のイベントをきっかけに処理を自動実行できる機能です。 CI/CDでは、テスト、ビルド、デプロイなどを自動化するためによく利用されます。

GitHub Actionsとは

GitHub Actionsは、リポジトリに対する操作をきっかけに、あらかじめ定義した処理を実行する自動化サービスです。 設定はYAMLファイルで管理し、リポジトリ内に置くことでコードと一緒に履歴管理できます。

「GitHubで起きた出来事に反応して、決めておいた作業を実行する仕組み」

なぜCI/CDで利用されるのか

GitHubと連携しやすい

push、pull request、tag作成など、開発で使うイベントをそのまま自動化の起点にできます。

設定をコード化できる

Workflowファイルをリポジトリで管理できるため、変更履歴やレビュー対象にできます。

小さく始めやすい

最初はテスト実行だけ、慣れてきたらビルドやデプロイも追加するように段階的に導入できます。

Core Terms

基本用語

Runner

Runnerとは

Runnerは、Workflowに書かれた処理を実際に実行するマシンです。 たとえばubuntu-latestを指定すると、Ubuntu環境のRunner上でコマンドが実行されます。

Workflow

Workflowとは

Workflowは、自動化したい一連の処理をまとめた設定です。 .github/workflows/配下にYAMLファイルとして配置します。

Job

Jobとは

Jobは、Workflow内で実行される処理のまとまりです。 テスト用Job、ビルド用Job、デプロイ用Jobのように目的ごとに分けられます。

Step

Stepとは

Stepは、Jobの中で順番に実行される1つ1つの作業です。 コマンドを実行するrunや、既存Actionを使うusesを指定できます。

Event

Eventとは

Eventは、Workflowを開始するきっかけです。 代表例はpush、pull_request、手動実行のworkflow_dispatchです。

用語の関係

Event pushやpull requestが発生
Workflow 自動化の設定全体が起動
Job テストやビルドなどのまとまりを実行
Step コマンドやActionを順番に実行
Runner 実行環境として処理を動かす

ポイント

Workflowを読むときは、まずonで「いつ動くか」を確認し、次にjobsで「何をするか」を見ます。 最後に各stepsを上から順に追うと、処理の流れをつかみやすくなる。
