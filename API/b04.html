APIレスポンスの構成要素について

01

この章の目標

  • 下記の APIレスポンスの構成要素 について説明できるようになる
    • ステータスコード
    • レスポンスヘッダ
    • レスポンスボディ
  • 仕様書に記載されているレスポンスの期待値からテストケースを想定できるようになる
02

概要

項目名 概要 例
ステータスコード レスポンスの状態を示す 3桁の数字 + ステータス
レスポンスヘッダ レスポンス自体に関する情報 認証情報など
レスポンスボディ APIから返却されたデータ JSON形式であることが多い
03

HTTPステータスコード

  • APIレスポンスの状況を表す情報
  • レスポンスが成功したかどうか、失敗している場合は何が原因かをおおよそ判断できる

ステータスコードとは

APIへリクエストを送信すると、

リクエスト
↓
API処理
↓
レスポンス

の流れで結果が返ってくる。

その結果を表すのが ステータスコード である。

イメージ

例えば

ユーザー情報を取得してください

というリクエストを送った場合、

成功

200 OK

↓

正常に処理できました

失敗

404 Not Found

↓

指定された情報が見つかりません

代表的なHTTPステータスコード

HTTPステータスコード - Wikipedia も参照のこと

ステータスコード 概要 エラー原因
200 リクエスト成功 -
400 リクエスト失敗 リクエストボディやパラメータが期待通りではない
401 認証失敗 ヘッダ設定不備などにより認証できていない
403 権限不備 エンドポイントへアクセスする権限がない
404 リソースが存在しない URLやパラメータのタイプミス
502 通信失敗 リクエスト先が停止しているなど

よく使うステータスコードの覚え方

コード 覚え方 例
200 成功 ユーザー情報取得成功
商品登録成功
400 リクエスト内容がおかしい 必須項目が未入力
パラメータが不正
401 認証できない パスワード間違い
トークン未設定
403 権限がない 管理者専用APIへ一般ユーザーがアクセス
404 見つからない URL間違い
存在しないデータを取得
502 通信先の問題 接続先サーバー停止
通信エラー
04

レスポンスヘッダ

  • レスポンス自体に関する情報

参考：https://developer.mozilla.org/ja/docs/Glossary/Response_header

レスポンスヘッダとは

レスポンスの説明情報。

リクエストのヘッダと同様に

key : value

形式で表現される。

主な用途

例：

レスポンスのデータ形式

レスポンスサイズ

認証情報

キャッシュ設定

など。

イメージ

レスポンスボディが

荷物

なら、

レスポンスヘッダは

荷物の説明書

のような役割を持つ。

05

レスポンスボディ

  • APIから返却されたデータ
  • JSON形式であることが多い

レスポンスボディとは

APIの処理結果そのもの。

例：

{
    "id": 1,
    "name": "Taro"
}

イメージ

リクエスト

ユーザー情報を取得してください

↓

レスポンスボディ

{
    "id": 1,
    "name": "Taro"
}

つまり、

実際に欲しい情報

がレスポンスボディに入っている。

06

演習問題

01. HTTPステータスコードについて

下記のようなリクエストを送信した場合、

それぞれどのようなステータスコードが想定されるか、理由も添えて考える。

1. 誤ったパスワードでAPIリクエストを送信

2. URLに誤字が含まれる状態でAPIリクエストを送信

3. リクエストボディの要素が欠損した状態でリクエストを送信

4. 管理者権限が必要なAPIに一般アカウントでリクエストを送信

この問題で確認したいこと

APIテストでは、

リクエスト内容
↓
どのステータスコードが返るか

を予測できることが重要である。

例えば

パスワード間違い
↓
401

や

URL間違い
↓
404

のように、

状況から原因を推測できるようになることを目指す。

07

まとめ

  • ステータスコード はAPI処理結果を表す
  • レスポンスヘッダ はレスポンスに関する説明情報
  • レスポンスボディ はAPIから返ってくる実際のデータ
  • APIテストではステータスコードとレスポンス内容を確認することが重要
  • ステータスコードからエラー原因をおおよそ推測できる
