APIリクエストの構成要素について

01

この章の目標

  • 下記の APIリクエストの構成要素 について説明できるようになる
    • メソッド / URL / エンドポイント
    • ヘッダ
    • パスパラメータ / クエリパラメータ
    • リクエストボディ
  • 仕様書に記載されているパスパラメータやクエリパラメータから、どのようなURLが組み立てられるかを想像できるようになる
02

概要

項目名 概要 例
メソッド そのリクエストで行う操作の種別 GET / POST / DELETE など
URL リクエストを送るリソース先 https://example.com/api/my/path/...
ヘッダ リクエスト自体に関する情報 "Content-Type": "Application-Json"
パラメータ リクエスト先に関するオプション パスパラメータ、クエリパラメータなど
リクエストボディ リクエスト先に送信するデータ JSON形式の記載が多い
03

メソッド

  • そのリクエストで行う操作の種別
メソッド名 操作 例
GET リソースやデータの取得 ユーザIDを指定し、該当ユーザの詳細情報を取得するAPI
POST リソースやデータの作成 ユーザ名やパスワードを指定し、新規ユーザを作成するAPI
DELETE リソースやデータの削除 ユーザIDを指定し、該当ユーザを削除するAPI
PUT / PATCH リソースやデータの更新 ユーザIDを指定し、該当ユーザの情報を変更するAPI
  • 動作確認的にAPIを触る場合は、既存データへの影響が少ない GET メソッドを優先して使用するとよい
04

URL

  • リクエストを送るリソース先

[備考] リクエストURLの読み方

https://example.com/api/user/{id} というURLを考える。

項目 例 詳細
ドメイン https://example.com APIの管理元
パス /api/user/{id} 各機能の住所
05

エンドポイント

  • メソッド と URLのパス をまとめて エンドポイント と呼称する
  • どのAPIを操作しているかを述べたいときは エンドポイント 単位で呼称する
  • URLのパスだけでは機能を特定できないため
機能 エンドポイント
ユーザ作成エンドポイント POST /api/user
ユーザ削除エンドポイント DELETE /api/user/{id}

補足

ここは「URLのパスだけでは足りない」という点が大事です。

同じ /api/user/{id} でも、GET と DELETE では意味が違います。

06

ヘッダ

  • リクエストに関するタグ情報
    • key:value のフォーマットで示される

主なヘッダの例

https://www.echoapi.jp/blog/untitled-34/ も参照のこと

ヘッダ名 概要 例
Content-Type リクエストボディのデータ様式 application/json / text/plain
Content-Length リクエストデータの容量 -
Authorization 認証関連のヘッダ情報 Basic XXXXX
User-Agent ユーザ（クライアント）側の情報 -

補足

ヘッダは「このリクエストはどんなものか」をサーバーに伝えるための情報です。

たとえば「JSONを送ります」「ログイン済みです」などを伝えます。

07

パラメータ

  • リクエストに関するオプション
  • パスパラメータ と クエリパラメータ の2種類がある

パスパラメータ

  • URLのパスに含まれるパラメータ。下記で用いられることが多い
  • リソースの直接指定
  • 必須パラメータ
例：ユーザ検索API
　https://example.com/user/{userId}
　　- userIdを定めれば該当ユーザも一意に定まる
　　- userIdがなければユーザ検索として機能しない = 必須パラメータ
  • パス と パスパラメータ の違いについて
  • API仕様書で文字列が決まっているのが パス
  • ユーザ側で値を決められるのが パスパラメータ

例：QualityForwardAPI / テストサイクル取得API

https://cloud.veriserve.co.jp/api/v2/test_phases/{test_phase[id]}/test_suite_assignments/{test_suite_assignments[id]}/test_cycles

  • {...} で括られているものがパスパラメータ
  • そうでないものはパスの一部

補足

パスパラメータは「場所そのものを指定する値」です。

たとえば「どのユーザか」をIDで決めるようなときに使います。

クエリパラメータ

  • URLの末尾に ?{key}={value} の形で含まれるパラメータ。下記で用いられることが多い
  • リソースの範囲指定 / 絞り込み
  • 任意パラメータ
例：ユーザ一覧返却API
　https://example.com/userList?name=tanaka&lower_age=20
    - tanaka の部分一致と20歳以上という検索範囲指定・絞り込みをかけている
    - 各クエリパラメータがなくても一覧返却として機能する

補足

クエリパラメータは「条件を追加する値」です。

たとえば「名前で絞る」「年齢で絞る」などの検索条件に使います。

08

リクエスト確認の補足

メソッドエラーの例

# 送信したリクエスト
POST https://api.example.com/api/user/1234

# 受信したレスポンス
400
{
    message:"invalid method"
}

POST https://api.example.com/api/user/1234 に対して invalid method が返却された場合、HTTPメソッドがAPI仕様と一致していない可能性がある。

このような場合は、API仕様書で対象エンドポイントに指定されているメソッドを確認し、GET / POST / PUT / DELETE などの指定を見直す必要がある。

補足

URLが正しく見えても、メソッドが違うだけでエラーになることがある。APIテストでは「どこに送るか」だけでなく「何の操作として送るか」も確認する。

URL（ドメイン / パス）の指定例

@あなた さん

お疲れさまです。
テスト対象は下記となります。

https://api.example.com/api/user/
@あなた さん

お疲れさまです、一点追記です！
上記のURLは本番環境なのでユーザーに影響が出る場合があります。
下記開発環境ドメインでの操作をお願いいたします

https://api.example-develop.com

最初に提示された https://api.example.com/api/user/ は本番環境のURLである。

あとから開発環境ドメインとして https://api.example-develop.com が指定された場合、ドメイン部分を開発環境に置き換え、パス部分は維持してリクエスト先を組み立てる。

https://api.example-develop.com/api/user/

補足

URLは「ドメイン」と「パス」に分けて考えると確認しやすい。本番環境と開発環境を取り違えると、実データに影響する可能性があるため注意が必要である。

エンドポイント確認の例

# API仕様書
GET /api/user/{id}
POST /api/user
DELETE /api/user/{id}
# メンバーからのチャット

@あなた さん

お疲れさまです。
/api/user/{id}にリクエストを送信したのですがエラーになりました。
ダブルチェックをお願いいたします。

この例では、メンバーから /api/user/{id} というパスだけが共有されている。

しかしAPI仕様書には、同じパスを使う GET /api/user/{id} と DELETE /api/user/{id} が存在するため、パスだけではどの操作を行ったのか特定できない。

エラーを確認する場合は、送信したHTTPメソッド、実際に指定したURL、パスパラメータに入れた値、ヘッダやボディの有無も合わせて確認する。

補足

エンドポイントは メソッド + URLのパス で判断する。同じパスでも、GET と DELETE では意味が大きく異なる。

クエリパラメータを含むURLの例

# APIテスト仕様書

GET https://api.example.com/api/user/search
下記のクエリパラメータを指定して送信する
region : Japan
user_status : premium

クエリパラメータはURLの末尾に ? を付け、その後に key=value の形で指定する。

複数のクエリパラメータを指定する場合は、2つ目以降を & でつなぐ。

https://api.example.com/api/user/search?region=Japan&user_status=premium

補足

? はクエリパラメータの開始位置、& は複数条件の区切りを表す。検索条件や絞り込み条件を指定するときによく使われる。

09

まとめ

  • メソッド は「何をするか」
  • URL は「どこに送るか」
  • エンドポイント は「メソッド + URLのパス」
  • ヘッダ は「リクエストの説明情報」
  • パラメータ は「条件や指定」
  • パスパラメータ は「場所を指定する値」
  • クエリパラメータ は「検索条件を追加する値」
  • リクエストボディ は「送るデータ本体」
