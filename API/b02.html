03

リクエストとレスポンスとは

APIを利用するときは、

リクエスト
↓
レスポンス

の流れで通信が行われる。

リクエスト（Request）

APIに対して

○○をしてください

とお願いすること。

例：

ユーザー情報を取得してください

商品一覧を取得してください

ユーザーを登録してください

レスポンス（Response）

APIがリクエストを受け取った後に返す結果。

例：

ユーザー情報

商品一覧

登録結果

など。

04

イメージ

アプリ
↓
リクエスト
「ユーザー情報を取得してください」

API
↓
レスポンス
「ユーザー情報はこちらです」

07

[余談] APIテストの実装イメージ

極論を言うと、APIテストは次の内容を確認している。

事前操作 期待結果
仕様書通りのリクエストを送る 仕様書通りのレスポンスが返ってくる
08

APIテストの考え方

APIテストは

正しいリクエスト
↓
正しいレスポンス

を確認する作業である。

09

「仕様書通りのリクエスト」とは？

例えば以下のような内容を確認する。

  • メソッドに○○を指定する
  • URLに○○を指定する
  • ヘッダに Authorization: ○○ を指定する
  • リクエストボディに○○を記述する

など。

補足：メソッドとは

APIに対してどのような操作を行うかを表す。

例：

メソッド 内容
GET 取得
POST 作成
PUT / PATCH 更新
DELETE 削除

補足：Authorizationヘッダとは

認証情報を送るためのヘッダ。

例：

Authorization: Bearer xxxxx

ログイン済みユーザーであることをAPIへ伝える。

補足：リクエストボディとは

APIへ送るデータ本体。

例：

{
    "name": "Taro",
    "email": "taro@example.com"
}
10

「仕様書通りのレスポンス」とは？

例えば以下のような内容を確認する。

  • ステータスコードが○○である
  • レスポンスボディが仕様書通りの構造である
  • レスポンスボディの値の型が正しい
  • リクエストで送信した値が正しく返っている

など。

補足：ステータスコードとは

API処理結果を表す番号。

例：

コード 意味
200 成功
400 リクエスト不正
401 認証失敗
403 権限不足
404 リソースが存在しない

補足：レスポンスボディとは

APIから返ってくるデータ本体。

例：

{
    "name": "Taro",
    "email": "taro@example.com"
}

補足：データ型とは

値の種類のこと。

例：

型 例
文字列（string） "Taro"
数値（integer） 100
真偽値（boolean） true / false
