APIリクエスト/レスポンスについて

APIにおけるリクエストとレスポンス

01

この章の目標

  • APIを経由して情報を取得する仕組みを理解できるようになる
02

リクエストとレスポンス

リクエストとレスポンスとは

APIでは、

リクエスト
↓
レスポンス

という流れでデータのやり取りを行う。

(HTTP/HTTPS)リクエストとは

  • APIに対して「○○をしてほしい」と依頼すること
  • クライアント（ブラウザやアプリ）からAPIへ送信する

例：

ユーザー情報を取得してください

商品一覧を取得してください

ユーザーを登録してください

HTTP / HTTPSとは

リクエストを送る際に利用される通信ルール（プロトコル）。

Web APIでは主に

HTTP
HTTPS

が利用される。

(API)レスポンスとは

  • APIがリクエストを受け取った後に返却する情報
  • APIからの「返事」

例：

ユーザー情報

商品一覧

登録結果

など。

イメージ

アプリ
↓
リクエスト
「ユーザー情報を取得してください」

API
↓
レスポンス
「ユーザー情報はこちらです」
03

APIテストの基本的な考え方

APIテストは非常にシンプルに表現すると、

事前操作 期待結果
仕様書通りのリクエストを送る 仕様書通りのレスポンスが返ってくる

APIテストとは

つまり、

正しいリクエスト
↓
正しいレスポンス

になっていることを確認する作業である。

04

仕様書通りのリクエストとは

リクエストには様々な構成要素がある。

例えば、

  • メソッドに○○を指定する
  • URLに○○を指定する
  • ヘッダに Authorization: ○○ を指定する
  • リクエストボディに○○を記述する
  • その他必要な情報を設定する

など。

例

GET
https://api.example.com/user/123

なら

GET

も

https://api.example.com/user/123

も仕様書通りである必要がある。

05

仕様書通りのレスポンスとは

レスポンスについても様々な確認を行う。

例えば、

  • ステータスコードが○○である
  • レスポンスボディが仕様書通りの構造である
  • レスポンスボディの値の型が正しい
  • リクエストで送信した値が正しく返っている

など。

例

リクエスト

{
    "name": "Taro"
}

を送信した場合、

レスポンスも

{
    "name": "Taro"
}

のように仕様通りであることを確認する。

ステータスコードの確認例

200

↓

成功
404

↓

対象が存在しない
06

APIテストを細かく見ると

実際のテストケースでは、

仕様書通りのリクエスト

と

仕様書通りのレスポンス

をさらに細かく分解して確認している。

例えば、

URLは正しいか

認証情報は正しいか

パラメータは正しいか

レスポンス内容は正しいか

などを1つずつ確認する。

07

まとめ

  • APIは リクエスト と レスポンス のやり取りで動作する
  • リクエストは「APIへのお願い」
  • レスポンスは「APIからの返事」
  • APIテストは「仕様書通りのリクエストを送った時に、仕様書通りのレスポンスが返るか」を確認する
  • 実際のテストではリクエストとレスポンスを細かい要素に分解して確認する
